Plan de tests End-to-End — Parcours Administrateur RH

Projet : Billed — Application de gestion des notes de frais  Périmètre : Parcours utilisateur "Administrateur RH"
Auteur : Mathys Henneron  Date : Mai 2026  Version : 1.0

1. Introduction

1.1 Objectif du document

Ce document décrit le plan de tests End-to-End (E2E) pour le parcours Administrateur RH de l'application Billed. Il vise à garantir le bon fonctionnement de bout en bout du parcours utilisateur, depuis la connexion administrateur jusqu'à la validation ou au refus d'une note de frais soumise par un employé.

1.2 Périmètre

Sont couverts par ce plan de tests :

  • L'authentification d'un utilisateur de type Admin (formulaire Administration)
  • L'affichage du Dashboard et le filtrage des notes par statut (En attente / Validé / Refusé)
  • Le dépliage des sections et l'affichage des cartes de notes de frais
  • La consultation du détail d'une note de frais (formulaire à droite)
  • La visualisation du justificatif via la modale dédiée
  • La validation (Accepter) et le refus (Refuser) d'une note de frais
  • La gestion des erreurs API (404, 500) et des états vides
  • La déconnexion

Sont hors périmètre : le parcours employé (couvert par un plan dédié), la création de notes de frais, et l'API back-end (testée séparément).

1.4 Prérequis

  • Le back-end Billed est lancé (npm run run:dev)
  • Le front-end est servi par live-server
  • La base de données contient au moins une note de frais par statut (en attente, acceptée, refusée)
  • Les cookies / localStorage sont vides au début de chaque scénario

2. Stratégie de test

Les tests E2E sont exécutés manuellement dans un navigateur avec Chrome DevTools ouvert pour vérifier les requêtes réseau et l'état de l'application. Chaque scénario suit la structure suivante :

ChampDescription
IDIdentifiant unique du scénario (ex : E2E-ADM-01)
TitreNom court du scénario
Pré-conditionsÉtat de l'application avant exécution
ÉtapesActions utilisateur, numérotées
Résultat attenduComportement attendu de l'application
TypePassant (cas nominal) / Non-passant (cas d'erreur)
PrioritéHaute / Moyenne / Basse

Critère global de réussite : 100% des scénarios passants doivent réussir sans régression. Les scénarios non-passants doivent afficher un retour utilisateur explicite (message d'erreur, blocage de l'action) sans rupture de l'application.

3. Scénarios de tests — Cas passants

E2E-ADM-01 — Connexion d'un administrateur RH

Priorité : Haute  Pré-conditions : L'utilisateur n'est pas connecté. Page de login affichée.

#Action
1Saisir admin@test.tld dans le champ "Votre email" du formulaire Administration
2Saisir admin dans le champ "Mot de passe"
3Cliquer sur le bouton "Se connecter" du formulaire Administration

Résultat attendu :

  • L'utilisateur est redirigé vers #admin/dashboard
  • Le localStorage contient un objet user de type Admin et de statut connected
  • La page "Validations" est affichée avec la grande icône Billed (data-testid="big-billed-icon")
  • L'icône "mail" du menu vertical n'est pas active (parcours admin différent de l'employé)

E2E-ADM-02 — Affichage du Dashboard avec les notes de frais

Priorité : Haute  Pré-conditions : Administrateur connecté ; la base contient des notes dans plusieurs statuts.

#Action
1Naviguer vers #admin/dashboard

Résultat attendu :

  • Trois sections sont affichées avec le compteur de notes par statut : En attente (n), Validé (n), Refusé (n)
  • Le compteur reflète exactement le nombre de notes filtrées via filteredBills en excluant les comptes de test
  • La zone "Validations" à droite affiche la grande icône Billed
  • Une requête GET /bills 200 OK est visible dans l'onglet Network

E2E-ADM-03 — Dépliage d'une section et affichage des cartes

Priorité : Haute  Pré-conditions : Administrateur sur le Dashboard. Au moins une note "en attente" existe.

#Action
1Cliquer sur la flèche (arrow-icon1) de la section "En attente"

Résultat attendu :

  • La flèche pivote (transform: rotate(0deg) → état déplié)
  • Les cartes des notes de frais en attente apparaissent dans status-bills-container1 (une carte par note, data-testid="open-bill<id>")
  • Chaque carte affiche : prénom/nom de l'employé, libellé, montant, date, type
  • Un second clic sur la flèche replie la section sans affecter les autres (cf. correctif du conflit de dépliage entre sections)

E2E-ADM-04 — Ouverture du détail d'une note en attente

Priorité : Haute  Pré-conditions : Section "En attente" dépliée, au moins une carte affichée.

#Action
1Cliquer sur une carte open-bill<id>

Résultat attendu :

  • Le formulaire dashboard-form s'affiche à droite, à la place de la grande icône Billed
  • Les champs sont remplis (type, nom, date, commentaire, montant TTC, TVA, justificatif)
  • Les boutons btn-refuse-bill-d et btn-accept-bill-d sont visibles (statut "pending")
  • Le champ commentary2 est éditable
  • Un second clic sur la même carte referme le formulaire et réaffiche la grande icône (big-billed-icon)

E2E-ADM-05 — Validation d'une note de frais (Accepter)

Priorité : Haute  Pré-conditions : Formulaire d'une note "en attente" ouvert.

#Action
1Saisir un commentaire dans le champ commentary2 (facultatif)
2Cliquer sur le bouton "Accepter" (btn-accept-bill-d)

Résultat attendu :

  • Une requête PATCH /bills/<id> avec status: "accepted" et commentAdmin est envoyée (visible dans Network)
  • La navigation repart vers #admin/dashboard
  • Le compteur "Validé" est incrémenté de 1, "En attente" est décrémenté de 1
  • La grande icône Billed (big-billed-icon) remplace le formulaire

E2E-ADM-06 — Refus d'une note de frais (Refuser)

Priorité : Haute  Pré-conditions : Formulaire d'une note "en attente" ouvert.

#Action
1Saisir un commentaire dans le champ commentary2 (facultatif)
2Cliquer sur le bouton "Refuser" (btn-refuse-bill-d)

Résultat attendu :

  • Une requête PATCH /bills/<id> avec status: "refused" et commentAdmin est envoyée
  • La navigation repart vers #admin/dashboard
  • Le compteur "Refusé" est incrémenté de 1, "En attente" est décrémenté de 1
  • La grande icône Billed remplace le formulaire

E2E-ADM-07 — Visualisation du justificatif

Priorité : Moyenne  Pré-conditions : Formulaire d'une note de frais ouvert (statut quelconque).

#Action
1Cliquer sur l'icône œil icon-eye-d du formulaire

Résultat attendu :

  • Une modale Bootstrap (modaleFileAdmin) s'ouvre, centrée, taille modal-lg
  • Le titre de la modale est "Justificatif"
  • L'image du justificatif est insérée dans .modal-body (<img> avec l'URL data-bill-url)
  • La modale peut être fermée via le bouton "×" ou un clic à l'extérieur

E2E-ADM-08 — Déconnexion

Priorité : Moyenne  Pré-conditions : Administrateur connecté.

#Action
1Cliquer sur l'icône "Déconnexion" (layout-disconnect) en bas du menu vertical

Résultat attendu :

  • Le localStorage est vidé (clé user supprimée)
  • L'utilisateur est redirigé vers la page de login (/)
  • Le mot "Administration" est de nouveau affiché (formulaire admin visible)

4. Scénarios de tests — Cas non-passants

E2E-ADM-09 — Connexion admin avec champs vides

Priorité : Moyenne  Pré-conditions : Utilisateur sur la page de login.

#Action
1Laisser les champs admin-email-input et admin-password-input vides
2Cliquer sur "Se connecter" du formulaire Administration

Résultat attendu :

  • La validation HTML5 native (required) bloque la soumission
  • Aucun appel API n'est effectué
  • L'utilisateur reste sur la page Login, sur le formulaire form-admin

E2E-ADM-10 — Connexion admin avec email au mauvais format

Priorité : Moyenne  Pré-conditions : Utilisateur sur la page de login.

#Action
1Saisir pasunemail dans le champ admin-email-input
2Saisir azerty dans le champ admin-password-input
3Cliquer sur "Se connecter" du formulaire Administration

Résultat attendu :

  • La validation HTML5 du champ type="email" bloque la soumission
  • L'utilisateur reste sur la page Login (form-admin toujours visible)
  • Aucune redirection ni appel API ne se produit

Ce scénario valide le correctif des sélecteurs incorrects sur le formulaire de login admin : avant correction, les data-testid renvoyaient à des inputs employé et la validation HTML5 admin était inopérante.

E2E-ADM-11 — Erreur 404 lors du chargement du Dashboard

Priorité : Moyenne  Pré-conditions : Le back-end est mocké pour renvoyer une 404 sur GET /bills.

#Action
1Se connecter en tant qu'administrateur
2Naviguer vers #admin/dashboard

Résultat attendu :

  • Le composant ErrorPage est rendu
  • Le message contenant Erreur 404 est affiché à l'écran
  • Les sections du Dashboard ne sont pas affichées
  • Le menu vertical reste accessible (déconnexion possible)

E2E-ADM-12 — Erreur 500 lors du chargement du Dashboard

Priorité : Moyenne  Pré-conditions : Le back-end est mocké pour renvoyer une 500 sur GET /bills.

#Action
1Se connecter en tant qu'administrateur
2Naviguer vers #admin/dashboard

Résultat attendu :

  • Le composant ErrorPage est rendu
  • Le message contenant Erreur 500 est affiché à l'écran
  • Aucune section n'est dépliée

E2E-ADM-13 — Dashboard sans aucune note de frais

Priorité : Basse  Pré-conditions : La base ne contient aucune note de frais (ou seules des notes des comptes de test sont présentes).

#Action
1Se connecter en tant qu'administrateur
2Cliquer sur la flèche d'une section

Résultat attendu :

  • Les compteurs des sections affichent (0)
  • Aucune carte (open-bill<id>) n'apparaît dans status-bills-container
  • Aucune erreur n'est levée dans la console

7. Conclusion

Ce plan de tests E2E couvre 13 scénarios sur le parcours administrateur RH : 8 cas passants et 5 cas non-passants. Il garantit que les fonctionnalités principales (authentification, consultation des notes de frais, validation, refus, visualisation du justificatif, déconnexion) ainsi que la gestion des erreurs sont validées de bout en bout avant la mise en production.

L'exécution complète du plan est estimée à 30 minutes par run en exécution manuelle. Une automatisation future avec Cypress ou Playwright est recommandée pour intégrer ces tests dans la CI/CD, notamment pour les scénarios de validation/refus qui sont les plus critiques côté métier.
